DESIGN SYSTEM (mandatory — full spec in LC76_Design_Guidelines.html, now v1.5)
- Fonts: Bebas Neue (display), Oswald / 'Oswald','Arial Narrow',sans-serif
  (body), IBM Plex Mono (labels/mono). The whole library is now on this single
  on-system set — no off-system faces anywhere.
- Reference doc tokens: --ink:#0d0e12, --paper:#dad2c0, --paper2:#edeae0,
  --amber:#b86800, --amber2:#e07800, --rule:#d4cfc4, --steeldk:#2a2f3a,
  --steelmd:#6a7080, --red-warn:#B91C1C, --green-ok:#059669
- Calculator tokens (warm palette): --clay, --ochre, --sand, --cream, --ink,
  --bark, --smoke
- Dark masthead + light content area pattern
- Sticky nav: masthead and TOC MUST be wrapped together in a single sticky
  container (position:sticky; top:0; z-index:100). Never separate.
- Content width: max-width:100% with percentage padding
  (padding: 28px 5% 80px). NEVER a fixed pixel max-width cap. Headers,
  footers, sticky nav always full browser width.
- TOC navigation: JS scrollIntoView / computed-offset scrollTo (not href
  anchors). Scroll offset must account for full sticky container height.
  IntersectionObserver for active-state highlighting. (DG v1.5 blesses both
  the inline onclick+scrollIntoView and the data-target+scrollTo patterns;
  the latter is preferred for new builds.)
- v1.5 mobile PWA pattern (MANDATORY, all 53 files conform): html
  {overflow-x:hidden; overflow-x:clip; overscroll-behavior-x:none} — on html,
  NEVER body (body breaks Safari sticky); a ::before slab on the sticky-nav
  unit (bottom:100%; height:100vh) with an opaque background; safe-area-inset-
  top padding; overscroll-behavior-x:contain on inner horizontal scrollers;
  a print guard resetting the sticky unit to position:static and suppressing
  the ::before slab.
- Print CSS: Design Guidelines Section 16 print standard — 7mm page margins,
  7.5pt body, no forced page breaks by default, break-inside:avoid on small
  elements only, break-inside:auto + thead repeat on large tables (15+ rows).
  Firefox: redefine CSS custom properties as hex with !important inside
  @media print — var(--token) does not resolve reliably in Firefox print.
- Calculator shared files (embedded in other pages) must have the index nav
  button removed so they remain standalone-shareable.
- 60-30-10 colour rule and WCAG AA compliance apply throughout.

WORKING CONVENTIONS
- File naming: LC76_Title_Case.html. No legacy patterns (hzj76r-, 1hz-,
  service-history-).
- Bar is the primary pressure unit on all displays; PSI is derived/secondary.
- Weights in kg, distances in km, pressures in bar (PSI shown alongside).
- R-numbering is authoritative — cross-check LC76_Library_Register.txt before
  building index updates; never assume the project file set reflects the live
  GitHub state.
- For large files (R4, R8, and any 80k+ HTML): start a fresh chat, upload
  the live HTML from GitHub, provide a change spec, execute surgical edits.
  Never rebuild from project-knowledge fragments.
- For structural or cross-file changes: propose first, get approval, then
  build. Flag the full required change map when a literal edit would create
  internal contradictions elsewhere in the document.
- Before batch-editing a UI element across files, ask Neil to identify a
  confirmed working file as the reference template. Never conclude an element
  is absent without careful full-file inspection.
- Update index.html AND LC76_Library_Register.txt when adding new documents.
- Service-worker cache versioning (Claude-owned — do NOT rely on Neil to
  remember this): whenever a library file is ADDED, REMOVED, or RENAMED,
  Claude bumps the CACHE constant in sw.js (e.g. 'lc76-library-v1' ->
  'lc76-library-v2') and includes the updated sw.js in the deliverables for
  that session. When adding a new document, sw.js must also be re-generated so
  the new file joins the offline precache list. Plain edits to existing files
  self-refresh via the worker's background revalidation and do NOT require a
  bump. Neil simply commits sw.js alongside the changed files — no decision
  needed. (Note: this convention only fires for changes made through Claude;
  files hand-edited and pushed to GitHub directly will not trigger it.)
- Automatic date bumps: on every edited document that carries a masthead
  "Last updated" line, bump it to the session date (files without such a line
  get nothing added — do not fabricate one). Bump index.html's header-meta
  "Last Updated" whenever any library file changes. Internal governance files
  (the audit HTML, the working papers) are NOT in the index or precache, so
  editing them triggers neither an index bump nor an sw.js bump.
- Monthly odometer update: if a session opens in a NEW calendar month and no
  odometer reading has been logged, ask Neil for the current reading before
  proceeding; then update LC76_Facts.txt §2 and propagate current-odometer
  references library-wide (NEVER touch historical event mileages — exhaust
  install 309,968; injection overhaul 310,412; etc. stay fixed). Current
  reading: 311,000 km (read Jul 2026).
- All country/terrain profiles are on GitHub (incl. Angola). The register's
  "NOT ON GITHUB" section is empty — nothing is intentionally excluded.
- Chrome extension get_page_content is unreliable — prefer web_fetch with a
  direct URL, or ask Neil to paste the URL.
- Supplier naming discipline: forward-looking check/action items must use
  generic terms ("the workshop," "a diesel/turbo workshop"). Historical
  service records may retain supplier names. Dirk Du Plessis / Dup Diesel
  is no longer the preferred turbo/fuel-system workshop.
- GPS coordinate policy: individual rock art panel locations (Tsodilo Hills),
  specific Himba village coordinates, black rhino sighting locations, and
  Maasai boma locations must never appear in public-facing documents.

SESSION EXECUTION & VALIDATION (every HTML session)
- Fetch SHA-pinned (see SOURCE OF TRUTH below), then edit surgically.
- Edits: Python str.replace with an anchor guard (assert count == 1) on every
  old string. No blind or global replacements.
- Validation gates before staging: div balance open == close (run a depth walk
  — final depth 0, min depth 0, no negative events — not just a tag count);
  node --check on every inline <script>; stale-term sweep to zero (documented
  exemptions stand); anchor/id uniqueness. For tag-balance repairs, walk spans
  too.
- Working papers (LC76_Library_Audit_WorkingPapers.xlsx): fetch live; edit
  append-only with openpyxl; NEVER run recalc.py; preserve the COUNTIF/COUNTIFS
  formula-occurrence baseline (currently 11); clone cell styles via
  copy.copy(src._style); verify testzip() == None; never load data_only=True
  (live formulas must survive).
- Known do-not-re-flag items (documented false positives): Service_Checklist
  has zero horizontal scrollers → vacuously compliant on the overscroll check;
  R5 carries two dead sticky selectors (.hdr and .masthead,.hero) from an
  earlier style swap — only the live .sticky-wrap is patched.

SOURCE OF TRUTH / PROJECT KNOWLEDGE POLICY
- GitHub is the single source of truth for all HTML documents. Project
  knowledge holds only the lightweight authoritative texts: Library
  Register, Facts.txt, Design Guidelines, Profile Template, briefing
  card, project instructions, and service xlsx. Never
  edit a document against a project-knowledge copy — always fetch live,
  SHA-PINNED, before editing: take the HEAD SHA from the repo Atom feed
  (github.com/Overlandslabs/Consumption/commits/main.atom — the first
  /commit/<40-hex>), then fetch
  raw.githubusercontent.com/Overlandslabs/Consumption/<SHA>/<filename>.
  Pinning to a SHA rather than /main/ guards against the repo moving
  mid-session. The live repo routinely runs AHEAD of the project file set and
  of any handover doc — always reconcile against the working papers / live
  HEAD at session start.

SURGICAL EDIT CHANGE-SPEC FORMAT
- Surgical edits must follow the standard change-spec format before
  execution: (1) target file and R-number, (2) exact old string to be
  replaced, (3) exact new string, (4) propagation map — any other files
  or register entries that require a matching change. Claude proposes the
  full spec; Neil confirms before any edit is made. (Pre-approved handover
  packs are specs in themselves — execute autonomously within their scope,
  surfacing only genuine decision gates with a recommended default.)
