LC76 Overland Series

Design Guidelines

iOS-first field tools — 60-30-10 colour structure, WCAG AA accessibility, Apple HIG compliance

1 — Philosophy & principles

Built for the field

These tools are used on dusty tracks in low light, one-handed, under pressure. Every decision must serve readability, speed, and trust. Aesthetics serve function — not the reverse.

The field testA user must be able to read any value while driving on corrugated dirt in bright sunlight. If it fails that test, it fails.
iOS-firstAll tools are PWAs on iPhone Safari. Design for 390px wide first, then adapt upward. Minimum touch target: 44×44pt per Apple HIG.
Brand identityThe LC76 aesthetic references the Southern African bush — Msasa copper, ochre dust, dark ironwood bark. This identity is fixed. The 60-30-10 rule and accessibility standards are applied within this identity.
No decorative complexityNo gradients on content. No shadows on text. No animations that delay access to data. Dark masthead + light content is the only layout pattern.

3 — 60-30-10 colour rule

Structure by proportion

The 60-30-10 rule governs how much of each colour group appears in any screen. It prevents visual noise and creates hierarchy that works in harsh field conditions.

60%Dominant
cream / white
30%Secondary
ink / bark
10%Accent
clay / ochre
60% — Dominant
--cream, --white
All card surfaces, content areas, input backgrounds. The visual resting state.
30% — Secondary
--ink, --bark
Masthead, vehicle bar, table headers, footer. The structural chrome.
10% — Accent
--clay, --ochre
Active terrain card, primary buttons, data values, status indicators.
Ochre as sub-accentWithin the 10% accent budget, ochre (--ochre) is used for data values, links, and informational highlights. Clay (--clay) is reserved for action: buttons, active states, and alerts requiring response.
4 — Contrast & WCAG AA

Accessibility ratios

All text must meet WCAG AA minimum: 4.5:1 for normal text, 3:1 for large text (18px+ or 14px+ bold). Status information must never rely on colour alone — always pair with a label or icon.

8 — Touch targets & HIG compliance

44pt minimum

Apple HIG requires all interactive elements to have a minimum touch target of 44×44 points. On standard iPhone (1x logical pixels = points), this is 44px. All buttons, checkboxes, sliders, and interactive cells must meet this.

44px
Button
minimum
☰
Icon button
minimum
Primary btn
Primary CTA
44px height
Slider track — full width, 44px tap zone

44px height
Slider touch zonesSliders themselves have a 4px visual track but the parent .sblock must be at minimum 44px tall so the user can reliably interact on a moving vehicle.
Table rowsInteractive table rows (leg rows with inputs) must be minimum 44px tall. Read-only rows can be 36px.
/* Enforce touch targets */
.btn, button, [role="button"] { min-height: 44px; }
.sblock { min-height: 44px; display: flex; flex-direction: column; justify-content: center; }
input[type="range"] { height: 44px; } /* tap zone, visual track via CSS */
9 — Dark / light hybrid model

Intentional hybrid

The tools use a deliberate hybrid — not full dark mode, not full light. The masthead and structural chrome use the dark palette; all data content uses the light palette. This is intentional and must not be changed.

Dark zones (30%)
Masthead
Vehicle bar
Table headers
Footer
Landing page background
Light zones (60%)
Cards & results
Conditions / sliders
Table rows
Input fields
Info bars
No system dark mode overrideDo not implement prefers-color-scheme: dark media queries. The hybrid model is fixed and does not respond to the OS dark mode toggle. The dark masthead is always dark; the content is always light.

12 — PWA & iOS compliance

Add to Home Screen ready

  • viewport-fit=cover — enables content to extend behind the notch / Dynamic Island
  • apple-mobile-web-app-capable: yes — enables full-screen mode when added to home screen
  • apple-mobile-web-app-status-bar-style: black-translucent — matches --ink background through status bar
  • theme-color: #1C1410 — browser chrome matches masthead on Android
  • manifest.json — defines app name, icon, start URL, display:standalone
  • Service worker — network-first — always fetches latest version; cache as offline fallback only
  • No prompt() or alert() — all interactions inline, no browser dialogs (blocked in standalone mode)
  • No horizontal overflow — overflow-x:hidden on html and body; tables in scroll wrappers
  • Safe area insets — use padding: env(safe-area-inset-*) for content near screen edges
<!-- Required meta tags — every tool -->
<meta name="viewport" content="width=device-width,initial-scale=1.0,viewport-fit=cover">
<meta name="apple-mobile-web-app-capable" content="yes">
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">
<meta name="apple-mobile-web-app-title" content="LC76 [Tool Name]">
<meta name="mobile-web-app-capable" content="yes">
<meta name="theme-color" content="#1C1410">
<link rel="manifest" href="manifest-[tool].json">
<link rel="apple-touch-icon" href="icon-[tool]-192.svg">

13 — Do / Don't

Common mistakes

✓ Do

Use token names: color:var(--bark)

Use 44px min touch targets on all interactive elements

Pair colour status with a label or icon

Use Oswald for all text — even numbers

Keep sliders full-width on mobile — never constrained to partial width

Show units on every value: km, L, km/L, bar, kg, m

Use onchange for number inputs that trigger recalc (prevents freeze on partial entry)

✗ Don't

Use raw hex values: #3D2B1A

Use prompt() or alert() — blocked in standalone PWA

Rely on colour alone for status — always add text

Use Inter, Roboto, Arial, or system fonts

Let content overflow horizontally — wrap tables in scroll containers

Use oninput on number fields that do heavy recalc — causes freeze mid-typing

Add animations that delay access to data values

}
